Technical overview

Sulkan runtime model

The current Sulkan source presents a focused runtime contract. Startup initializes shader-pack management, water and shadow pipelines, and renderer state. On client start, Sulkan verifies a Vulkan backend and bootstraps the selected pack state into the active resource-pack repository.

Workflow

Shader pack lifecycle

The manager behavior is structured enough to document as a stable user workflow.

1. Discovery

Sulkan scans the game `shaders` folder and classifies each entry as supported or unsupported. Supported entries may be directories or `.zip` files.

2. Normalization

Packs may ship as `assets/sulkan/shaders/...` directly, or as shorthand `shaders/...`, which Sulkan maps into the Sulkan namespace during staging.

3. Active pack staging

Selected external packs are rebuilt into a generated active resource pack named `sulkan_shaderpack_active` before being inserted into resource-pack selection.

4. Reload and invalidation

When the active pack changes, Sulkan triggers resource reloads and prompts shadow renderer refresh and level render-data reset where needed.

Water and shadows

Technical feature summary

These notes summarize what Sulkan exposes without mirroring the source files directly.

Terrain shadowing

Up to four cascades are budgeted, refreshed at different tick intervals, and blended by receiver distance. Quality presets and render distance influence the active count and effective coverage.

Entity shadowing

Entity receivers use a three-cascade terrain shadow stack plus a dedicated dynamic entity shadow depth source for short-range detail.

Wave timing

Water animation is driven by a uniform buffer updated once per frame with runtime wave time, allowing deterministic animated displacement and ripple logic in water shaders.

Custom celestials

Sulkan defines a custom orbital yaw for sun and moon direction, giving the shadow system an authored celestial orientation rather than vanilla-only assumptions.
